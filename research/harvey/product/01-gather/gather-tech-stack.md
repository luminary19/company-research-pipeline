### AREA 3 — AI / Agent Architecture (crown jewels)

- **"No custom orchestration" rule — every top-level thread is an agent; all features are Tool Bundles.** Three scaling principles verbatim: (1) "No Custom Orchestration: All new product features that live in Assistant are Tool Bundles, and every top-level thread interface is an agent." (2) "Capabilities are Tool Bundles." (3) "Eval Gates on Capabilities: Tool Bundles and system prompt upgrades must pass leave-one-out gates to be deployed." `[R1·C4]` (blog + ZenML LLMOps writeup corroborates)
  - Source: https://www.harvey.ai/blog/principles-that-helped-us-scale-agent-development ; https://www.zenml.io/llmops-database/scaling-agent-based-architecture-for-legal-ai-assistant
- **Tool Bundle definition (verbatim).** "An agent at Harvey is composed of a system prompt and a set of Tool Bundles. A Tool Bundle is an interface we designed that allows developers to package together new capabilities, which may be composed of multiple tools or sub-agents, into a single entity. For example, a file system Tool Bundle comprises a grep-like file search tool, a file open tool, and a semantic search tool... A drafting Tool Bundle comprises an editing tool and a drafting sub-agent." Bundles let feature teams "inject instructions into the main agent system prompt" and are "portable between different agents." `[R1·C3]`
  - Source: https://www.harvey.ai/blog/principles-that-helped-us-scale-agent-development
- **Eval gates = leave-one-out validation on benchmarks.** Guards against three failure modes named verbatim: "System Prompt → Tool Bundle conflicts," "Tool Bundle → System Prompt conflicts," and "Context rot." Each Tool Bundle owner maintains datasets + evaluators + thresholds "that fire if metrics drop below a certain score." Memorable engineer quote: "You're no longer merging unit-testable code, you're merging English." `[R1·C3]`
  - Source: https://www.harvey.ai/blog/principles-that-helped-us-scale-agent-development
- **Multi-model by design — routes across Anthropic, Google DeepMind, OpenAI (+ self-hosted open-source).** "Last year, Harvey went multi-model, expanding its platform to incorporate leading foundation models from Anthropic, Google DeepMind, and OpenAI." Task-to-model matching examples (as of Mar 2026): "Claude Opus 4.6... excels at agentic reasoning and deep multi-step legal analysis, while GPT-5.2 pairs strong analytical abilities with capability awareness"; for high-volume/low-latency surfaces like Vault, "models like Sonnet 4.6 and Gemini 3 Flash." Multi-model rationale: quality, reliability (structural redundancy/failover), and customer choice (workspace admins can disable providers; users pick a model in a selector). `[R1·C4]`
  - Source: https://www.harvey.ai/blog/why-harvey-is-multi-model-by-design
- **Why they built their own cloud agent runtime instead of using managed runtimes (Anthropic/OpenAI/AWS/Foundry/Google).** Three hard blockers: multi-model, zero data retention, cost. Key architecture line: "We built an abstraction layer that normalizes the harness, the sandbox, and the behavioral differences beneath a single interface, so that for everything above it, the choice of model is just a routing decision." Cost payoff: "empirically we see 3-5x cost reductions versus a frontier-only approach"; routes each task "to the most efficient model that meets the quality threshold, including open-source models we host ourselves." ZDR is architectural, not bolt-on: "customer data is not written into durable application storage by default... that disk is lifecycle-bound to the sandbox and automatically cleaned up as part of teardown." `[R1·C3]`
  - Source: https://www.harvey.ai/blog/why-we-built-our-own-cloud-agent-infrastructure
- **Spectre — internal collaborative cloud agent platform (the coding-agent equivalent, by Joey Wang & Gabe Pereyra, Apr 2026).** Core primitives verbatim: "durable runs, ephemeral workers" ("the durable object is the run record, not the worker process"); "Sandboxes as the execution boundary" (workers isolated, no ambient access to control plane, "durable state changes go back through the control plane"); "A harness, not just a wrapper" (assembles context, starts provider adapter, translates provider-native events into a stable internal shape); "Collaboration surfaces are part of the runtime" (a run can start in Slack, web, or CLI and all refer to the same durable run); "Scheduled Runs" via cron-based automations. Connects to GitHub, Datadog, Linear "through explicit boundaries." Security model maps to legal: "repositories and pull requests become matters and review workflows; diffs and execution history become document versions, provenance, and approval trails; sandbox boundaries and scoped credentials become ethical walls, client isolation, and controlled access." `[R1·C3]`
  - Source: https://www.harvey.ai/blog/building-spectre-internal-collaborative-cloud-agent-platform
- **Eval infrastructure = BigLaw Bench + LAB (Legal Agent Benchmark).** Multi-model matching is "informed by continuous evaluation through BigLaw Bench and expert preference testing from our Applied Legal Research team." LAB "shows clear separation by practice area and task type." `[R1·C4]`
  - Source: https://www.harvey.ai/blog/why-harvey-is-multi-model-by-design ; https://www.harvey.ai/blog/training-a-legal-agent-with-applied-compute
- **RAG design — LanceDB Enterprise in production; Postgres+PGVector for smaller/public-data projects.** "We primarily use LanceDB Enterprise in production because of its strength across latency, accuracy, ingestion throughput, scalability, data privacy, and hosting. Meanwhile, we still appreciate the simplicity and feature-completeness of Postgres for rapid development." Benchmarks cited: Postgres/PGVector "500K embeddings <2s P50"; LanceDB "15M rows w/ metadata filtering: <2s P50" and "massive-scale IVF-PQ index." Enterprise requirement: VectorDB "hosted within a private cloud environment and store both embeddings and source data in our customers' own secure storage (e.g., their private cloud buckets)." `[R1·C3]`
  - Source: https://www.harvey.ai/blog/enterprise-grade-rag-systems
- **Custom legal embeddings = voyage-law-2-harvey (partnership with Voyage AI).** Fine-tuned from `voyage-law-2` base on "more than 20 billion tokens" of US case law + expert-annotated exemplar questions. Result: "reduces the amount of irrelevant material returned in top results by nearly 25% compared to the next best off-the-shelf embedding models (e.g. Google's text-embedding-004 or OpenAI's text-embedding-3-large)... with 1/3 of the embedding dimensionality." Metrics used: NDCG@10 and Recall@100. `[R2·C4]` (Voyage AI blog, corroborated by Harvey blog)
  - Source: https://blog.voyageai.com/2024/07/31/harvey-partners-with-voyage-to-build-custom-legal-embeddings/ ; https://www.harvey.ai/blog/harvey-partners-with-voyage-to-build-custom-legal-embeddings
- **On-demand vision system for document images (Tom McCormick, Anna Zhang, Gary Lam, May 2026).** "An on-demand visual analysis tool that the agent invokes only when it needs to" — vision is a TOOL, not an indexing step. Pipeline: agent detects a visual question → "smart page finding" narrows a 500-page doc to 2-3 candidate pages in milliseconds via existing text-search infra → "high-fidelity rendering" by "a dedicated rendering service... separate from the main application" at high DPI → "structured visual reasoning" (reads chart axes, interpolates values between tick marks, distinguishes exact vs approximate, reports confidence levels). "Text-first, vision-second" gating. Cost economics: "Processing a single image is roughly 50x more expensive than generating a text response... our pipeline handled billions of images... 90% of those images are not actually necessary." Graceful degradation instead of hallucinating. `[R1·C3]`
  - Source: https://www.harvey.ai/blog/building-image-understanding-for-legal-documents
- **Post-trained legal model on GLM-5.1 via Applied Compute (Niko Grupen, Julio Pereyra, Gabe Pereyra et al., Jun 2026).** Full-parameter, fully asynchronous reinforcement learning on "Applied Compute Agent Cloud" / "AC2." Selected GLM-5.1 for strongest open-weight baseline. Results: rubric pass rate 0.853 → 0.913 and all-pass rate 0.059 → 0.126; "outperforms Opus 4.8 Max and GPT-5.5 xhigh" on rubric pass rate (second to Opus 4.8 Max on all-pass). Full stack described: grader alignment (GPT-5 Mini as grader, validated against a GPT-5.5 xhigh / Opus 4.8 Max "frontier consensus" agreeing on >95% of criteria), cost levers (batching criteria per grader call → "40x cost reduction at 16 criteria per group and 100x reduction when all criteria were grouped"), harness optimization (restrict tool-call token volume, repair malformed tool calls, better tool descriptions), and **compaction** ("~10% of base GLM-5.1 rollouts hit the context limit... 90th-percentile datapoint contains nearly 100,000 source-document tokens, and the largest exceed 200,000" — harness summarizes the transcript with the same weights + different system prompt, then continues from the summary). Learned behaviors flipped: artifact completeness (185 criteria), specificity/exactness (243), grounding/hallucination reduction (70), "despite not training against an explicit hallucination penalty." Future work named: "relevance-masked self-distillation" and "agentic router training." `[R1·C3]`
  - Source: https://www.harvey.ai/blog/training-a-legal-agent-with-applied-compute
- **Memory / context management is a first-class platform concern.** AI Platform team charter: "model routing, agent architecture, context management, evals"; "Build the platform-level systems for context management, session state, and memory that all of Harvey's agents and products rely on." `[R1·C4]`
  - Source: https://www.harvey.ai/company/careers/51fb953a-c494-4a09-8fe2-8d7268b863ec ; https://www.builtinsf.com/job/staff-software-engineer-ai-platform/9158607
- **"Data Factory" — autonomous multi-agent pipeline scaling knowledge sources from 6 to 60+ jurisdictions** (400+ legal data sources since Aug 2025). Notable architecture choice: instead of one agent per jurisdiction, "we treat domain-specific sources as parameterized tools... The result is a single reasoning system" (avoids sub-agent handoff losing conversation history). `[R1·C3]`
  - Source: https://www.harvey.ai/blog/using-agents-to-scale-harveys-knowledge-sources

  - Source: https://www.harvey.ai/security ; https://trust.harvey.ai/ ; https://www.harvey.ai/legal/security-addendum
- **Encryption: AES-256 at rest, TLS 1.2+ in transit; "no unencrypted pathways."** "When customers upload files, those files are transmitted over encrypted channels (TLS 1.2+) to blob storage within the customer's regional environment, and each file is encrypted at rest using an AES-256 key... decrypted only in memory." `[R1·C4]`
  - Source: https://www.harvey.ai/blog/how-harvey-manages-customer-data
- **Tenant isolation is enforced at the database/vector-store layer, not filtered after the fact.** "We partition that storage so that each workspace has its own dedicated, isolated footprint, backed by separate collections and storage. A query issued on behalf of one customer cannot see or reference another customer's vectors, because the database itself is segmented along tenant boundaries and tenant IDs rather than filtered after the fact." Also: "Customer data for each customer is logically separated to prevent any commingling." `[R1·C4]`
  - Source: https://www.harvey.ai/blog/how-harvey-secures-embeddings-at-scale ; https://www.harvey.ai/security
- **BYOK-style customer-controlled storage / customer-owned buckets.** Enterprise VectorDB "hosted within a private cloud environment and store both embeddings and source data in our customers' own secure storage (e.g., their private cloud buckets)... sensitive data never leaves the client's domain." Blog on cloud agent infra adds "sovereign deployments: the option to self-host their own cloud agent infrastructure through us, inside their own boundary." `[R1·C3]`
  - Source: https://www.harvey.ai/blog/enterprise-grade-rag-systems ; https://www.harvey.ai/blog/why-we-built-our-own-cloud-agent-infrastructure
- **Zero Data Retention (ZDR) + zero training on customer data — contractual and architectural.** "Harvey requires Zero Data Retention (ZDR) by model providers"; "Harvey contractually prohibits model providers from training on customer data." Multi-model post: "zero data retention, no human review of customer data, and no training on customer inputs." Custom firm models only "if you explicitly request it... a bespoke model is created exclusively for you, and your data is never used to train models used by other customers." `[R1·C5]`
  - Source: https://www.harvey.ai/security ; https://www.harvey.ai/blog/why-harvey-is-multi-model-by-design
- **Data residency: three processing regions + storage in 30+ countries.** "We offer three data processing locations: our Standard Region (US, EU, Switzerland), our EU Region (EU and Switzerland), and our Australian Region (Australia)... customers a myriad of choices about where their data is stored (with possible regions including over 30 different countries)."   - Source: https://www.harvey.ai/blog/debunking-myths-data-sovereignty-and-us-based-service-providers ; https://www.harvey.ai/security
- **Access-control model: RBAC, network segmentation, identity federation, least privilege, short-lived credentials, IaC-declared access.** "Role-based access controls, network segmentation, and identity federation enforce this separation." ## Gaps and not-found

1. **Interview loop / hiring process** — number of rounds, on-site format, question style, take-home vs live-coding. Not in R1 careers pages. Needs R3 (jobsbyculture / techinterview.org / Glassdoor / levels.fyi).
2. **"Azure OpenAI Service" by name** — confirmed cloud host is Microsoft Azure and providers are Anthropic/Google/OpenAI, but the specific managed "Azure OpenAI Service" product is not named in any source read. Unverified.
3. **Node.js runtime** — frontend confirmed as React/TypeScript/TailwindCSS PWA, but the JS server/runtime (Node.js) is not named verbatim.
4. **Hallucination decomposition / citation-verification as a discrete named system** — grounding/hallucination reduction is documented as an *emergent training outcome* in the GLM-5.1 post and as vision-system "graceful degradation," but a standalone "citation verification / hallucination decomposition" subsystem is not described in the sources read. Likely lives in a separate blog (e.g., "Building an Agentic Security Operations Center," referenced but not crawled).
5. **Datadog/Sentry/PagerDuty/Incident.io as *currently deployed*** vs. "desired experience" — these appear in job requirements, which strongly implies in-use but is technically a requirement statement, not a direct "we run X" claim. (Temporal, Redis, PgBouncer, LanceDB, Postgres, blob storage ARE direct deployment claims from eng blogs.)
6. **GLM-5.1 model in production** — the Applied Compute post reports a benchmark result ("strongest available model on LAB by rubric pass rate"); it explicitly frames production deployment as forward-looking ("as these systems move from benchmarks into production"), so treat the trained GLM-5.1 as a research result, not a confirmed production model.
